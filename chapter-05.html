Chapter 5 · Exam favourite

Data Resource Management

Treating data as a valuable company asset: how it is organised, how databases beat loose files, and how warehouses and data mining turn piles of records into real business insight.

Exam radar: this chapter comes up a lot. The reliable favourites are the data hierarchy, the file-processing problems, the relational model, and especially the difference between a data warehouse and data mining (and OLAP). They are all flagged below.
In plain English — the whole chapter as one story

Our running example: BrewHaus, your coffee chain. Every sale, every loyalty member, every stock count is data. This chapter is about treating that data as a real asset and getting value out of it, instead of letting it sit in messy piles.

1. Data is organised in a ladder

It builds up in steps: a single letter or digit (character), grouped into a fact like a name (field), grouped into one customer's full details (record), grouped into all customers (file), all linked together (database). Once you see the ladder, databases stop being mysterious.

2. Keeping data in separate files causes chaos

If the till, the loyalty app and the accounts each keep their own customer list, the same person is stored three times, the lists disagree, and nothing joins up. That mess is exactly what databases fix.

3. A database puts everything in one shared place

Instead of three separate lists, one shared database holds the data once, and a piece of software called a DBMS manages who reads and writes it. No duplication, everything consistent.

4. Data can be structured in a few ways, and one won

Of the ways to organise a database, the relational model, simple tables linked by a shared column, won because it is flexible and easy to query. It is what BrewHaus's systems almost certainly use.

5. You ask the database questions with a language called SQL

"Which customers bought nothing last month?" becomes a short SQL query. You do not have to write programs; you just describe what you want.

6. To analyse history, you build a warehouse

Day-to-day databases are busy running the shop. So you copy cleaned-up history into a separate data warehouse built for analysis, then mine it to discover patterns (like "people who buy oat lattes also buy almond croissants").

How the chapter connects: data climbs a hierarchy → loose files create redundancy and chaos → a shared database with a DBMS fixes that → the relational model became the standard way to structure it → SQL lets you ask it questions → and for deeper analysis you copy history into a warehouse and mine it for patterns. Data goes from raw fact to business knowledge.

Section 1 · exam favourite

The data hierarchy

When we say "data," what are the building blocks, from smallest to biggest?

Data is organised as a ladder, where each rung is built from the one below it:

  • Character — the smallest piece: a single letter, digit or symbol.
  • Field — a group of characters forming one attribute (a fact about something), like a name, a salary, a job title.
  • Record — all the fields describing one entity (one person, place, thing or event), like one customer's whole row of details.
  • File (or table) — a group of related records, like every customer.
  • Database — an integrated collection of logically related files and records, all linked together.
Two words you need: an entity is a thing you track (a customer, a product, a meter). An attribute is a characteristic of that entity (the customer's name, the product's price). A field holds one attribute; a record holds all the attributes of one entity.
Section 2 · exam favourite

Why loose files fail

What was so bad about the old way of just keeping separate files?

In traditional file processing, every application kept its own private files. At BrewHaus that would mean the till, the loyalty app and the accounts each storing their own customer list. That causes four classic problems:

ProblemWhat goes wrong
Data redundancyThe same data is duplicated across many files (the same customer stored three times).
Lack of data integrationData is scattered, so you cannot easily combine it to answer a question.
Data dependencePrograms are tied to the exact file layout, so changing the data breaks the programs.
Lack of integrity / standardisationThe copies disagree and follow no common standard, so the data cannot be trusted.

These four problems are exactly what the database approach was invented to solve.

Section 3

The database approach and the DBMS

So what does the database approach do differently?

It consolidates the data into one shared database that many programs can use, instead of many private files. Sitting between the users and that shared data is a piece of software called the database management system (DBMS).

The DBMS does three core jobs, easy to remember as Create, Maintain, Use:

  • Create new databases and database applications.
  • Maintain the quality of the data (add, update, correct, delete).
  • Use the data to give end users the information they need (querying).
Section 4 · exam favourite

Ways to structure a database

Is there only one way to organise the data inside a database?

No, there are five fundamental structures. The first two are mostly history; the third one won.

StructureHow it organises dataStrength / weakness
HierarchicalA tree; strict one-to-many (each child has one parent).Good for routine transactions; cannot handle many-to-many.
NetworkA web allowing many-to-many links.More flexible than hierarchical; poor at one-off (ad hoc) requests.
RelationalSimple tables linked by a shared column.Most widely used; easily answers ad hoc questions and is easy to maintain; slightly slower than the older two.
MultidimensionalData viewed as cubes with several dimensions.Great for analysis (powers OLAP, see below).
Object-orientedObjects that bundle data with the operations on it, and support inheritance.Good for complex data types and reuse.
Why relational won: it trades a little raw speed for huge gains in flexibility and ease of asking ad-hoc questions. For most businesses that is the better deal, which is why nearly every business database today is relational.
Section 5 · exam favourite

The relational model up close

What does "relational" actually look like?

Data sits in tables. In each table, a row is a record and a column is a field. The magic is that you can relate one table to another whenever they share a common column (a key). That shared column is how the database joins things up.

Section 6

Asking the database questions

How do you actually get answers out of a database?

You interrogate it, and you do not need to write a program. There are two main ways to ask, and both give you a screen display or a printed report:

  • A query language for an immediate answer to a one-off (ad hoc) question.
  • A report generator to lay out information as a formatted report.

The standard query language is SQL, and its basic shape is worth memorising:

SELECT … FROM … WHERE …
SELECT which columns you want, FROM which table(s), WHERE the condition that filters the rows. Example in plain words: "select the customer names, from the customer table, where last month's orders is zero."

Because writing SQL by hand is fiddly, most database tools let you build a query by pointing and clicking (called query-by-example), or even ask in near-plain English, and the tool writes the SQL for you. To narrow a search you combine terms with Boolean operators: AND, OR, NOT (for example, "lattes AND Cairo NOT decaf").

Section 7

The types of databases

Is a database just one big thing, or are there different kinds?

There are several kinds, each for a different purpose:

  • Operational databases — hold the live data running the business day to day (customers, inventory, payroll). Also called transaction or production databases.
  • Distributed databases — copies or parts stored across several locations. Faster and safer, but the copies must be kept in sync, by replication (find and apply each change everywhere, complex) or duplication (copy one master to all sites after hours, simpler).
  • External databases — data available from outside, often for a fee. Even Google is, in effect, a giant external database.
  • End-user databases — the small databases an individual builds for themselves.
  • Hypermedia databases — the web pages, images, audio and video of a website, linked by hyperlinks rather than by record keys.
  • Data warehouses — the big analysis store, covered next.
Section 8 · exam favourite

Data warehouses

Why copy data into yet another database just to analyse it?

Because your live operational databases are busy running the shop; hammering them with heavy analysis would slow them down. So you build a separate data warehouse: a central store of static data that has been extracted from other databases, then cleaned, transformed and catalogued, ready for analysis and decision support. A warehouse can be sliced into data marts, smaller subsets focused on one department or process (a marketing mart, a finance mart).

Data mining and OLAP

Once the warehouse is full, how do you find the gold in it?

For BrewHaus, mining might reveal that customers who buy oat lattes very often add an almond croissant, so you bundle them as a deal.

OLAP versus data mining (a favourite exam contrast). OLAP (online analytical processing) uses multidimensional "cubes" to let you slice and view data you already understand from many angles (sales by product, by region, by month) — you are answering questions you already have. Data mining goes further: it discovers patterns you did not know to ask about. OLAP shows you the data; mining finds the surprises in it.
Section 10

Building a database, and who owns it

How is a database actually planned and built?

Through a five-step development process that moves from the business idea down to the physical storage:

  1. Data planning — model the business processes.
  2. Requirements specification — capture what end users need.
  3. Conceptual design — draw it as a high-level model (an entity-relationship diagram showing the entities and how they relate).
  4. Logical design — translate that into the database's data model (for example, relational).
  5. Physical design — decide how and where the data is actually stored.

The person responsible is the database administrator (DBA), who guards the integrity and security of the data, uses a data definition language (DDL) to specify its structure, and records all those definitions in a data dictionary — a catalogue of "data about data," also called metadata.

Q1List the data hierarchy from smallest to largest.
Character → field → record → file (table) → database.
Q2What are the four problems of traditional file processing?
Data redundancy, lack of data integration, data dependence, and lack of data integrity/standardisation.
Q3Why did the relational model become dominant?
It stores data in simple tables linked by a common column, which makes it flexible and easy to query for ad-hoc questions, even though it is slightly slower than older structures.
Q4What is the basic shape of an SQL query?
SELECT (which columns) … FROM (which tables) … WHERE (the filtering condition).
Q5What is the difference between a data warehouse and data mining?
A data warehouse is the cleaned, static store of historical data built for analysis; data mining is the process of analysing that store to discover hidden patterns and turn them into business knowledge.
Q6How does OLAP differ from data mining?
OLAP slices and views data you already understand from many dimensions (answering questions you have); data mining discovers patterns you did not know to look for.
Q7Describe the data warehouse load path.
Capture data from operational and external sources, clean and transform it into a consistent form, then load it into the warehouse (and its data marts) for analysis. This is often called ETL.
